# Hydra Result: Composer → Claude Code 消息发送机制分析

## 调查结论

**当前 `\x1bv` (Meta+V) 机制可以工作，但存在根本性的架构问题。建议 Claude Code 也切换为 `bracketed-paste` 模式。**

---

## 1. `\x1bv` 方式的具体问题

### 1.1 机制原理

当前 Claude Code (`inputMode: 'paste'`) 的发送流程：
1. `clipboard.writeText(text)` — 将文本写入系统剪贴板
2. `ptyManager.write(ptyId, '\x1bv')` — 向 PTY master 写入 ESC + v (2 bytes: `0x1b 0x76`)
3. `await delay(120ms)` — 等待粘贴完成
4. `ptyManager.write(ptyId, '\r')` — 发送回车提交

`\x1bv` 在终端协议中是 **Meta+V**（macOS 上 Option+V）。这不是系统粘贴快捷键（Cmd+V），而是 Alt/Option 修饰符 + V。

### 1.2 为什么这个方法"能用"

Claude Code 基于 Ink 框架，Ink 使用 `setRawMode(true)` 进入原始模式。在原始模式下：
- stdin 逐字节读取，不经过行编辑
- `\x1b` + `v` 被 Ink 的 `useInput` 解析为 `{input: 'v', key: {meta: true}}`
- Claude Code 内部很可能注册了 Meta+V 快捷键处理器，检测到后从系统剪贴板读取内容

这个机制是一条 **间接路径**：PTY 字节 → Ink 解析按键 → Claude Code 处理 Meta+V → 读取系统剪贴板 → 插入文本。

### 1.3 具体风险

| 风险 | 说明 | 严重性 |
|------|------|--------|
| **剪贴板竞争** | 步骤 1 和 2 之间（或 2 执行期间），其他应用可能覆盖剪贴板内容 | **高** |
| **用户剪贴板被破坏** | 虽然有 snapshot/restore，但如果 restore 失败（已有测试覆盖），用户剪贴板丢失 | **中** |
| **平台依赖** | macOS 用 `\x1bv` (Meta+V)，其他平台用 `\x0016` (Ctrl+V)，不同 Claude Code 版本可能处理方式不同 | **中** |
| **Timing 敏感** | 120ms delay 是经验值，系统负载高时可能不够；也可能过长影响体验 | **低-中** |
| **图片发送更脆弱** | 图片通过剪贴板传递，失败时回退到路径文本，需要两次 paste 操作 | **中** |

## 2. 字节拆分的风险分析

### 2.1 PTY 写入原子性

`node-pty` 的 `write(data)` 方法最终调用 POSIX `write()` 系统调用向 PTY master fd 写入数据。对于 `\x1bv` 这 2 个字节：

- **写入端**：单次 `write()` 系统调用写入 2 字节，操作系统保证原子写入（PIPE_BUF 通常 4096 字节，远大于 2 字节）
- **读取端**：子进程通过 `read()` 从 PTY slave fd 读取。在 raw mode 下（`VMIN=1, VTIME=0`），`read()` 返回当前可用的任何字节

### 2.2 拆分场景

**理论上**，2 字节 `\x1bv` 拆分的可能性极低但非零：
- 正常情况下，2 字节在一次 `read()` 中完整到达
- PTY 内核缓冲区不会拆分如此小的数据
- 但在极端系统负载下，或进程调度不巧时，`read()` 可能只返回 `\x1b`

### 2.3 拆分后果

如果 `\x1b` 和 `v` 分开到达：

1. Ink 的输入解析器收到 `\x1b`
2. Ink 启动 escape sequence 超时检测（通常 ~100ms）
3. 如果 `v` 在超时前到达 → 正确解析为 Meta+V → 正常工作
4. 如果 `v` 在超时后到达 → `\x1b` 被解释为单独的 Escape 键，`v` 被解释为字母 'v'
5. 结果：Claude Code 收到一个 Escape 按键和一个字母 'v'，不会触发粘贴

**实际风险评估**：极低概率，但一旦发生会导致消息丢失且用户看到 `v` 字符出现在输入区。

## 3. Bracketed Paste 是否更适合 Claude Code

### 3.1 Ink 的 Bracketed Paste 支持

Ink 文档明确说明：
> "If multiple characters are pasted at once, the callback is triggered once with the entire pasted string as the `input` argument."

这个行为的实现依赖于 **bracketed paste mode**。当终端启用 bracketed paste 后：
- 粘贴的文本被 `\x1b[200~` 和 `\x1b[201~` 包裹
- Ink 检测到开始标记后缓冲所有输入，直到收到结束标记
- 然后将整段文本作为一个 `input` 参数传给 `useInput` 回调

Ink 在启用 raw mode 时，很可能向 terminal 发送 `\x1b[?2004h` 来启用 bracketed paste mode。Claude Code 作为 Ink 应用，天然支持接收 bracketed paste 序列。

### 3.2 对比分析

| 维度 | `paste` (当前 \x1bv) | `bracketed-paste` |
|------|----------------------|-------------------|
| **文本传递** | 间接：剪贴板 → Meta+V → 读剪贴板 | 直接：文本嵌入 PTY 字节流 |
| **剪贴板依赖** | 必须使用系统剪贴板 | 完全不需要 |
| **竞争条件** | 存在剪贴板竞争窗口 | 无竞争条件 |
| **用户剪贴板** | 需要 snapshot/restore，可能丢失 | 不影响 |
| **字节拆分容错** | 2 字节拆分 → 完全失败 | 包裹标记内文本即使分批到达也能正确缓冲 |
| **多行文本** | 依赖 Claude Code 正确处理剪贴板中的多行 | Bracketed paste 天然支持多行，不会被解释为多次回车 |
| **Timing** | 需要 120ms delay 等待处理 | 写入即到达，delay 仅作安全余量 |
| **图片路径** | 可以，但每张图片都占用剪贴板 | 路径作为文本直接嵌入 |
| **Codex 已验证** | — | Codex 已使用此模式，测试已覆盖 |

### 3.3 结论

**Bracketed paste 明显更适合 Claude Code。** 原因：
1. 消除了对系统剪贴板的依赖，避免竞争条件
2. 文本直接通过 PTY 传递，路径更短、更可靠
3. Ink 框架原生支持 bracketed paste 解析
4. 与 Codex 的实现一致，减少维护成本
5. 代码更简单 — 不需要 clipboard snapshot/restore 逻辑

## 4. 具体修复建议

### 4.1 将 Claude Code 切换为 `bracketed-paste` 模式

在 `src/terminal/cliConfig.ts` 中修改 claude 配置：

```typescript
claude: {
    type: "claude",
    // ...
    composer: {
      supportsComposer: true,
      allowedStatuses: INTERACTIVE_STATUSES,
      inputMode: "bracketed-paste",       // 改为 bracketed-paste
      supportsImages: true,
      pasteKeySequence: () => "",          // 不再需要
      imageFallback: "image-path",         // 保留：图片仍作为路径发送
      pasteDelayMs: 120,
    },
},
```

### 4.2 验证其他 `paste` 模式的终端

当前使用 `inputMode: 'paste'` 的终端还有：
- **kimi** — 使用 `\x0016` (Ctrl+V)，需确认 Kimi CLI 是否支持 bracketed paste
- **gemini** — 使用 `\x0016` (Ctrl+V)，同上
- **opencode** — 使用 `\x0016` (Ctrl+V)，同上

建议逐一验证这些 CLI 是否也支持 bracketed paste，如果支持则统一切换。

### 4.3 考虑长期淘汰 `paste` inputMode

如果所有目标 CLI 都支持 bracketed paste，可以：
1. 移除 `paste` 模式的整个代码路径（clipboard snapshot/restore、pasteKeySequence 等）
2. 只保留 `type`（shell/lazygit/tmux 等简单终端）和 `bracketed-paste` 两种模式
3. 这会大幅简化 `composer-submit.ts` 中的逻辑

### 4.4 风险与回退

- **风险**：如果 Claude Code 某个版本不支持 bracketed paste，文本会被当作普通逐字符输入，可能触发意外行为
- **验证方法**：在实际 Claude Code PTY 中发送 `\x1b[200~test\x1b[201~\r`，观察是否正确接收 "test" 并提交
- **回退**：保留 `paste` 模式代码，仅修改配置默认值，必要时可快速回退
