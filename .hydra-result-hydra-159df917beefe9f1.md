## Issues found

### 1. High: Group 3 introduced an unresolved module-resolution regression in the shared shortcut helper
- Change group: `1a2e1c4` (`src/hooks/shortcutTarget.ts`, `src/stores/shortcutStore.ts`)
- The refactor moved `hasPrimaryModifier` into `src/hooks/shortcutTarget.ts` and imported it from `src/stores/shortcutStore.ts`. That new cross-file import is currently unstable across the project's two toolchains:
  - With the current HEAD (`46d5b09`), `src/stores/shortcutStore.ts` imports `../hooks/shortcutTarget` without an extension, which makes the Node test runner fail with `ERR_MODULE_NOT_FOUND`.
  - The earlier follow-up `32ff2c0` changed the import to `../hooks/shortcutTarget.ts`, but that conflicts with the repo's TypeScript config because `allowImportingTsExtensions` is not enabled, so `tsc --noEmit` fails instead.
- This means the merge did not resolve the root cause; it created a path that passes one environment by breaking the other.
- Impact:
  - Current relevant test coverage is red.
  - The shortcut refactor is not portable between the app bundler and the Node-based test runner.
- Recommended fix:
  - Pick one import style and make both toolchains support it, or move the shared helper into a location/module pattern that both bundler resolution and `node --experimental-strip-types` resolve consistently.

### 2. Medium: Group 2 still has an async focus-steal race after terminal switching
- Change group: `b56eb46` (`src/components/ComposerBar.tsx`)
- The post-submit focus logic now skips refocusing the Composer for `type` terminals, but it still decides that based on the `targetTerminal` captured before `await window.termcanvas.composer.submit(request)`.
- If the user submits to a bracketed-paste terminal and then switches focus via shortcut to a `type` terminal while the submit request is in flight, the old callback resumes with the stale terminal type and still runs `requestAnimationFrame(() => textareaRef.current?.focus())`.
- That re-steals DOM focus from the newly focused xterm and partially defeats the purpose of the focus-management fix.
- Recommended fix:
  - After the await, re-read the current focused/target terminal from store-derived state before deciding where focus should land.
