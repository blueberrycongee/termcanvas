# H1: Does `\x1bv` (Meta+V) actually trigger paste in Claude Code?

## Evidence FOR: Yes, it works — but indirectly

Claude Code is built on Ink (React for CLIs). Ink's `useInput` hook operates in raw mode and delegates to `parseKeypress()` to interpret raw stdin bytes. The parser uses `metaKeyCodeRe` to detect ESC+letter combinations:

```
metaKeyCodeRe = /^(?:\x1b)([a-zA-Z0-9])$/
```

When `\x1bv` arrives as a single chunk, Ink's parser:
1. Matches ESC + `v` via `metaKeyCodeRe`
2. Sets `key.meta = true`, `key.name = 'v'`
3. The `useInput` hook strips the leading `\x1b`, passes `input = "v"` with `key.meta = true`

Claude Code then receives `(input="v", key={meta: true})` — **Meta+V**. Claude Code almost certainly has a `useInput` handler that checks for Meta+V (or Option+V on macOS) to trigger paste-from-clipboard, since this is the standard macOS Option+V shortcut and Claude Code explicitly documents Ctrl+V (Linux/Windows) vs Option+V (macOS) for paste.

**TermCanvas's cliConfig confirms this pattern:**
- macOS: `pasteKeySequence = "\u001bv"` (ESC + v = Meta+V / Option+V)
- Linux/Windows: `pasteKeySequence = "\u0016"` (Ctrl+V)

## Evidence AGAINST: Fragile mechanism

The mechanism has **no protocol-level guarantee**. It relies on:
1. Claude Code's Ink app having a `useInput` handler that maps Meta+V to paste
2. The two bytes `\x1b` + `v` arriving as a single read() call
3. No other Ink component intercepting the ESC before the paste handler runs

**Critical finding**: Ink's `useInput` sets `key.escape = true` when `keypress.name === 'escape'`. If the bytes are split (see PTY atomicity section below), Ink would see a solo ESC first (name=escape), then a literal 'v' — **paste would not trigger**.

## Conclusion

`\x1bv` does trigger paste in Claude Code when delivered atomically. The mechanism is: raw bytes → Ink parseKeypress → Meta+V key event → Claude Code's paste handler reads system clipboard → inserts text. But it is **fragile** due to PTY byte splitting.

##### H5: Should Claude Code use bracketed-paste mode like Codex?

## Evidence: Claude Code DOES support bracketed paste

**Key finding**: Claude Code (via Ink) supports the `usePaste` hook, which enables bracketed paste mode (`\x1b[?2004h`). This is confirmed by:

1. **GitHub Issue #3134**: Claude Code enables bracketed paste mode but fails to disable it on exit, leaving terminals corrupted. This proves Claude Code's Ink app DOES activate `usePaste` / bracketed paste mode.

2. **Ink's `usePaste` implementation**: When active, it calls `setBracketedPasteMode(true)`, which writes `\x1b[?2004h` to stdout. Pasted text arrives wrapped in `\x1b[200~...\x1b[201~` and is emitted as a 'paste' event, delivered to the handler as a complete string.

3. **GitHub Issue #13183**: Reports that Claude Code's bracketed paste handling can hang with accessibility/voice-to-text tools, confirming Claude Code processes bracketed paste sequences.

## Recommendation: YES — TermCanvas should switch Claude Code to bracketed-paste mode

The Codex adapter already uses `inputMode: "bracketed-paste"`, which writes `\x1b[200~text\x1b[201~` directly to PTY. This approach:
- **Bypasses clipboard entirely** — no `clipboard.writeText()`, no race conditions with other apps
- **Is protocol-level** — Ink's paste event system handles the bracketed sequence natively
- **Delivers text atomically** — the entire message content arrives in one paste event
- **No Meta+V indirection** — removes the fragile keystroke simulation

Since Claude Code's Ink app already has `usePaste` active (proven by issue #3134), sending `\x1b[200~text\x1b[201~` directly to the PTY should work the same way as for Codex.

# PTY Atomicity: Can `\x1bv` be split?

## Yes — PTY writes are NOT guaranteed atomic

### POSIX semantics
- `write()` to a PTY master returns the number of bytes written. There is **no atomicity guarantee** for PTY writes comparable to pipe writes under `PIPE_BUF`.
- `read()` on the slave side can return fewer bytes than were written: "It is not an error if [the number of bytes read] is smaller than the number of bytes requested" (POSIX read(2)).
- The TTY line discipline sits between master and slave, copying data asynchronously. The kernel documentation states: "Data written to the slave will be available at the master promptly, but may not be available immediately."

### Known splitting scenarios

1. **CPU contention / scheduling**: If the reading process (Claude Code) is descheduled between the kernel delivering the first byte and buffering the second, it may see two separate `read()` results.

2. **TTY line discipline processing**: The line discipline processes bytes and may introduce timing gaps, especially under load.

3. **Documented in real software**:
   - **Microsoft Terminal Issue #4037**: Escape sequences split across multiple reads cause parsing failures. Arrow keys (`\x1b[A`) received as two separate reads.
   - **Micro editor Issue #3668**: Bracketed paste sequences split across reads — terminal sends `\x1b[200~` in one chunk and the rest + `\x1b[201~` in another.
   - **Foot terminal**: Known to split bracketed paste into 2 separate messages.

4. **For `\x1bv` specifically**: This is only 2 bytes. While less likely to split than longer sequences, under CPU load or when the PTY buffer happens to flush between bytes, the slave side could see:
   - Read 1: `\x1b` → Ink interprets as **ESC** (cancel/escape key)
   - Read 2: `v` → Ink interprets as **literal 'v'** character typed

### Consequence of splitting
If `\x1bv` splits into `\x1b` then `v`:
- Ink's `parseKeypress` sees a lone `\x1b` → `key.name = 'escape'`, `key.escape = true`
- Then sees `v` → `input = 'v'`, no meta flag
- **Paste never triggers**. The text in the clipboard is never inserted. The `\r` (Enter) sent 120ms later submits nothing or submits the literal 'v'.

# Recommendation: Most likely root cause

The **most likely root cause** of intermittent Composer failures with Claude Code is **PTY byte splitting of `\x1bv`**, combined with the fragility of the clipboard-based paste mechanism.

### Contributing factors (ordered by likelihood):

1. **PTY byte splitting** (HIGH): `\x1bv` splits into ESC + v under load. Claude Code sees an escape key press and ignores the subsequent 'v'. Paste never fires. The 120ms delay and subsequent `\r` submits nothing (or submits garbage).

2. **Clipboard race condition** (MEDIUM): Between `clipboard.writeText()` and Meta+V triggering paste, another process could overwrite the clipboard. macOS clipboard is a shared global resource.

3. **Ink focus/component state** (LOW): If Claude Code's input component that handles Meta+V is not focused or active when the keystroke arrives, the paste handler may not fire.

### Recommended fix:

**Switch Claude Code's `inputMode` from `"paste"` to `"bracketed-paste"`** in `cliConfig.ts`. This:
- Eliminates PTY byte splitting as a failure mode (content is delivered via protocol, not keystroke simulation)
- Eliminates clipboard as a shared-state dependency
- Aligns with the proven approach already working for Codex
- Leverages Claude Code's existing `usePaste` support (confirmed by GitHub issues)

The change would be minimal:
```typescript
claude: {
  // ...
  composer: {
    inputMode: "bracketed-paste",  // was "paste"
    // pasteKeySequence becomes unused
    // pasteDelayMs still useful between multiple image pastes
  }
}
```

### Caveats:
- Bracketed paste has its own edge cases (Issue #13183 shows hangs with accessibility tools)
- Very long messages may need chunking if PTY buffer limits apply to bracketed paste content
- Should test with multiline text containing escape-like characters
