# Hydra Task Result

## Files Changed

- **electron/session-watcher.ts** — Added logging, fallback polling, and completion guard

## Changes

### Root cause
`fs.watch` on macOS can miss file change events, especially when writes use atomic rename patterns (write to tmp file, then rename). When the `fs.watch` event is missed, the `checkTurnComplete` function never runs, and the terminal status stays stuck at 'waiting'.

### Fix
1. **Fallback polling**: After 30 seconds of watching, starts polling every 10 seconds via `setInterval` as a safety net. If `fs.watch` works normally, the polling never fires (the `completed` guard prevents double-notification).

2. **Completion guard**: Added `completed` boolean flag and `onCompleted()` wrapper to prevent duplicate callbacks from both `fs.watch` and the poll timer.

3. **Extracted `tryCheck` helper**: Both `fs.watch` debounce handler and poll timer use the same `tryCheck(source)` function for mtime + `checkTurnComplete` logic.

4. **`[SessionWatcher]` logging**: Added `console.log` with `[SessionWatcher]` prefix at all key points:
   - `watch()` called (sessionId, type, filePath)
   - `resolveSessionFile` returning null
   - `fs.watch` event received (event type, filename)
   - `checkTurnComplete` result (source, sessionId, completed)
   - Completion notification fired
   - Fallback polling started
   - `unwatch()` called

5. **Poll timer cleanup**: `unwatch()` now clears both `clearTimeout` and `clearInterval` on the poll timer, and the `WatchEntry` interface includes `pollTimer`.

## Tests

## Unresolved
- None. The fallback polling approach is defensive — if `fs.watch` works normally, the poll timer is either not started (completion within 30s) or immediately sees `completed=true` and stops.
