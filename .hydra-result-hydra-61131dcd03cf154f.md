Issues found
- High: `electron/usage-collector.ts:284` to `electron/usage-collector.ts:306` overcounts Codex usage for sessions that cross a day boundary. `total_token_usage` is cumulative for the whole session, but the collector keeps only the last in-range snapshot and never subtracts the session total at the start of the day. If a session starts before midnight and continues into the target day, the first in-day snapshot already includes prior-day tokens, so the target day is inflated.
- High: `electron/usage-collector.ts:83` to `electron/usage-collector.ts:96` computes the UTC filter window incorrectly for any non-UTC timezone, and `electron/usage-collector.ts:321` to `electron/usage-collector.ts:322` uses that window for all filtering. `new Date(\`${dateStr}T00:00:00\`)` already represents local midnight, so subtracting `tzOffsetHours` shifts the range a second time. In UTC+8, `2026-03-18` becomes `2026-03-17T08:00:00Z` instead of `2026-03-17T16:00:00Z`. In DST zones it is also wrong because the code uses the current offset, not the target date's offset.
- High: `electron/usage-collector.ts:7` to `electron/usage-collector.ts:13`, `electron/usage-collector.ts:75` to `electron/usage-collector.ts:80`, and `electron/usage-collector.ts:224` price Claude usage against model keys that do not match the actual session data. Real Claude logs use versioned IDs like `claude-opus-4-6-20251001` and `claude-haiku-4-5-20251001`; these miss the pricing table and fall back to `default`, which prices Haiku and Sonnet traffic as Opus-tier cost.
- Medium: `electron/usage-collector.ts:202` to `electron/usage-collector.ts:240` deduplicates Claude records only after applying the day filter. If the same `message.id` has an earlier partial entry before midnight and the final entry after midnight, each day can count a different version of the same message, so cross-midnight turns can be double-counted across daily reports.
- Medium: `electron/usage-collector.ts:334` to `electron/usage-collector.ts:338` and `electron/usage-collector.ts:352` to `electron/usage-collector.ts:355` use the current machine offset to localize `mtime`. That is usually fine in fixed-offset zones, but it is not correct for historical dates in DST regions because a January query in New York is checked with March's offset.
- Medium: `electron/usage-collector.ts:324` to `electron/usage-collector.ts:358` does not meaningfully reduce blocking. `setImmediate()` only defers the work to the next event-loop turn; once the callback starts, the synchronous `readdirSync`, `statSync`, and `readFileSync` loops still block the Electron main process until the scan finishes.
- Medium: `src/components/UsagePanel.tsx:220` to `src/components/UsagePanel.tsx:225` and `src/stores/usageStore.ts:24` to `src/stores/usageStore.ts:38` allow overlapping polls. There is no in-flight guard, so a slow `usage:query` can still be running when the next 60s interval fires, stacking extra main-process scans.
