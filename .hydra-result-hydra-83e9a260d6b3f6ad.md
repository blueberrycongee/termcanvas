## Issues found

### Critical

1. `electron/usage-collector.ts:195-225` and `electron/usage-collector.ts:247-287`
   - The parser counts every Claude `message.usage` line and every Codex `token_count` event as a separate billable record, but both real log formats emit repeated snapshots for the same logical turn. In actual `~/.claude` data, the same `message.id` appears on multiple lines with repeated/intermediate usage snapshots. In actual `~/.codex` data, identical `last_token_usage` payloads are emitted more than once for the same turn. This will overcount tokens, calls, project totals, model totals, and cost, sometimes by a large margin.

2. `electron/usage-collector.ts:300-305`, `electron/main.ts:560-563`, `electron/usage-collector.ts:315-319`, and `electron/usage-collector.ts:331-335`
   - Daily filtering is tied to a hardcoded UTC+8 default and then mixed with UTC file mtimes. `collectUsage()` is always called without the machine timezone, so users outside UTC+8 get the wrong day boundaries and wrong bucket placement. Separately, the quick mtime filter compares a UTC `toISOString().split("T")[0]` date to a local `YYYY-MM-DD`, which can skip files that were modified shortly after local midnight but still belong to the requested local day.

3. `electron/usage-collector.ts:7-12`, `electron/usage-collector.ts:73-79`, and `electron/usage-collector.ts:278-284`
   - Cost math is materially wrong for Codex sessions and likely wrong for Claude Opus pricing. Codex records are forced to model `"codex"` and then fall through to the default Claude pricing table, so OpenAI usage is priced as if it were Claude Opus. Current official OpenAI pricing for `codex-mini-latest` is `$1.50` input / `$0.375` cached input / `$6.00` output per 1M tokens, not the `$5 / $0.50 / $25` default used here. Anthropic’s current Opus pricing is also much higher than the table in code. Because the panel’s main output is cost, this breaks the feature’s headline metric.

### Warning

1. `electron/usage-collector.ts:99-126`
   - Claude file discovery is incomplete. It only scans `~/.claude/projects/*/*.jsonl` and `~/.claude/*.jsonl`, but real local data also exists under paths like `~/.claude/projects/<project>/<session>/subagents/*.jsonl` and `~/.claude/transcripts/*.jsonl`. Those sessions are silently omitted, so totals can be incomplete even before the parsing bugs above.

2. `electron/main.ts:560-563` plus most of `electron/usage-collector.ts`
   - The renderer IPC invokes a fully synchronous home-directory scan on the Electron main thread. `readdirSync`, `statSync`, and `readFileSync` are used across every candidate session file, and each file is read fully into memory. On large Claude/Codex histories this will block the main process and can freeze the UI when the panel opens and on every 60-second refresh.

### Suggestion

1. `src/components/UsagePanel.tsx:177-190`, `src/components/UsagePanel.tsx:278-287`, and `src/toolbar/Toolbar.tsx:87-100`
   - The panel still has avoidable UX/accessibility gaps: the section title `"Tokens"` is hardcoded instead of localized, the icon-only toolbar button has no explicit accessible name/state beyond `title`, and the resize handle is mouse-only with no keyboard affordance. These are not blockers, but they make the feature less usable and less consistent with the rest of the app.

2. `src/types/index.ts:139-149`
   - Renderer-side `ProjectUsage` and `ModelUsage` types only expose `cost` and `calls`, while the collector actually returns token totals as well. TypeScript accepts the extra fields structurally, but the source-of-truth types are now divergent between main and renderer, which makes future maintenance and testing easier to get wrong.

3. `tests/`
   - There is no automated coverage for this feature. The repo currently has no tests for usage-file discovery, deduplication, timezone filtering, pricing, or the `UsagePanel` rendering states. Given the parsing complexity and the amount of silent `catch {}` logic, this feature needs targeted fixture-based tests.
