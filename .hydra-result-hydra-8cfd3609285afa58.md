Files changed and why
- `src/terminal/TerminalTile.tsx`: removed the observer-driven `fitAddon.fit()` path and replaced it with a fit effect keyed to the tile's React-controlled `width`/`height`/`minimized` state. This prevents streaming output from triggering unintended xterm resizes.

Issues found
- `xterm.write()` itself does not directly change `ydisp`; `CoreTerminal.write()` only queues data into `WriteBuffer`.
- During parse, output can change `ydisp` through `BufferService.scroll()`. When the user has scrolled up, normal new output keeps `ydisp` stable while `ybase` grows. If scrollback is trimming because the buffer is full, xterm decrements `ydisp` to keep the same visible text stable.
- `fitAddon.fit()` can change `ydisp` indirectly because it calls `terminal.resize()` whenever rows/cols change. In xterm's `Buffer.resize()` and reflow code, row/column changes adjust `ybase`/`ydisp`.
- Claude-style TUI rewrite sequences like cursor-up, erase-line, and rewrite do not themselves decrement `ydisp`; they mainly move the cursor or mutate line contents. The parser paths that actually move `ydisp` are scrolling and resize/reflow paths.
- The most plausible root cause in this app is the `ResizeObserver` around `fitAddon.fit()`. It allows DOM/layout churn during streaming to trigger background terminal resizes, and xterm's resize logic then nudges `ydisp` while the user is scrolled up.
