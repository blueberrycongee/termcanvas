## Core Architecture Summary

### Terminal Status State Machine (TerminalTile.tsx:275-415)

```
PTY created → "running"
        │
        ├── PTY output received → "active"
        │       │
        │       └── 15s no output → "waiting"
        │               │
        │               └── more output → "active" (cycle)
        │
        ├── session watcher fires turn-complete → "completed"
        │       (only from "active" or "waiting")
        │
        └── PTY exit → "success" (exit 0) / "error" (exit ≠ 0)
```

### Composer allowedStatuses for Claude Code

Claude uses `INTERACTIVE_STATUSES`:
```typescript
["running", "active", "waiting", "completed", "success", "idle"]
```

This is the **broadest** set — it includes every status except `"error"`. The Composer will allow sending in **all** of these states.

### How Composer Sends Input

For Claude Code (`inputMode: "paste"`):
1. Writes text to system clipboard
2. Sends paste key sequence (⌘V on macOS / Ctrl+V on Linux) to PTY
3. Sends `\r` (Enter) to PTY

This is a **blind fire-and-forget** — there is no acknowledgment from Claude Code that the input was received or accepted.

## Edge Case Analysis: Message Loss Scenarios

### 1. Claude Code Is Actively Generating a Response

| Attribute | Value |
|-----------|-------|
| **Terminal status** | `"active"` (PTY is receiving output) |
| **Composer allows send?** | ✅ Yes (`"active"` ∈ INTERACTIVE_STATUSES) |
| **Claude Code accepts input?** | ❌ No — Claude Code's TUI input field is not active during generation |
| **What happens** | The paste key sequence (⌘V) is sent to the PTY. Claude Code's Ink-based TUI is rendering output and does not have an active text input. The paste is either silently dropped by the TUI framework or interpreted as a hotkey (e.g., `v` might trigger a view action). The `\r` (Enter) is similarly dropped or triggers an unintended action. |
| **Result** | **Message silently lost.** No error returned to user. |
| **Probability** | **HIGH** — this is the most common mismatch. Any time Claude is mid-response, the terminal shows "active" but Claude doesn't accept input. |
| **Trigger** | User types follow-up while Claude is still responding. |

### 2. Claude Code Is Showing a Permission Prompt (Y/N)

| Attribute | Value |
|-----------|-------|
| **Terminal status** | `"waiting"` (output stopped >15s while waiting for user) or `"active"` (prompt just appeared) |
| **Composer allows send?** | ✅ Yes |
| **Claude Code accepts input?** | ⚠️ Yes, but only Y/N/Esc — not arbitrary text |
| **What happens** | The pasted text replaces the expected Y/N input. If the text starts with `y` or `Y`, it **approves the permission request unintentionally**. If it starts with `n` or `N`, it denies. Otherwise, Claude Code likely ignores or handles it unpredictably. The `\r` confirms whatever was entered. |
| **Result** | **Unintentional permission approval/denial + message lost.** The user's actual message is consumed by the permission prompt, and the follow-up question never reaches Claude. |
| **Probability** | **MEDIUM** — occurs when Claude requests tool use approval and the user sends a new message before seeing/acknowledging the prompt. |
| **Trigger** | User composes a message while Claude is waiting for permission confirmation. Particularly dangerous with auto-switching focus. |
| **Severity** | **CRITICAL** — this can cause unintended code execution (if the message starts with 'y'). |

### 3. Claude Code Is Executing a Tool (e.g., running bash, reading files)

| Attribute | Value |
|-----------|-------|
| **Terminal status** | `"active"` (tool output is streaming) |
| **Composer allows send?** | ✅ Yes |
| **Claude Code accepts input?** | ❌ No — TUI is in tool execution display mode |
| **What happens** | Same as Scenario 1 — paste and Enter are sent to a non-receptive TUI. |
| **Result** | **Message silently lost.** |
| **Probability** | **HIGH** — tool use is frequent during Claude Code sessions. Long-running tools (bash commands, file reads) keep the terminal in "active" for extended periods. |
| **Trigger** | User sends message while Claude is executing tools. |

### 4. Race Condition: Status Transition Timing

| Attribute | Value |
|-----------|-------|
| **Terminal status** | Transitioning between states |
| **Composer allows send?** | Depends on timing |
| **What happens** | Several race windows exist: |

**4a. "active" → "waiting" transition (15s timer)**
- The 15-second timer has not yet fired but Claude has finished and is ready for input.
- Terminal shows "active" but Claude is actually ready.
- Composer sends → **Message delivered successfully** (lucky case).

**4b. "waiting" → "active" transition (new output)**
- Claude starts a new response just as the user hits send.
- Composer checks status, sees "waiting" (valid), begins clipboard+paste sequence.
- By the time the paste key reaches Claude, it's already generating.
- **Message lost** due to ~120ms paste delay + clipboard operations.
- Probability: **LOW** but nonzero.

**4c. "active" → "completed" transition (session watcher)**
- The session watcher debounces by 300ms. During this window, the JSONL file has the completion signal but the terminal status hasn't updated yet.
- Composer sees "active", sends → **Message may be delivered** if Claude's TUI has already switched to input mode.
- Probability: **LOW** — the 300ms debounce creates a small window.

### 5. PTY Process Exited but Status Not Yet Updated

| Attribute | Value |
|-----------|-------|
| **Terminal status** | Still `"active"` or `"waiting"` (exit event hasn't propagated) |
| **Composer allows send?** | ✅ Yes |
| **Claude Code accepts input?** | ❌ Process doesn't exist |
| **What happens** | The `composer:submit` handler checks `ptyManager.getPid(request.ptyId)` and returns `target-not-running` error. This is the **only** server-side guard. |
| **Result** | **Error returned to user** — this case is actually handled correctly. |
| **Probability** | **LOW** — the PTY exit event is synchronous in node-pty and fires promptly. |

### 6. Claude Code Is in Compact Mode / Scrollback Review

| Attribute | Value |
|-----------|-------|
| **Terminal status** | `"waiting"` (no output while user scrolls) |
| **Composer allows send?** | ✅ Yes |
| **Claude Code accepts input?** | ⚠️ Depends — if in scrollback/compact mode, input handling varies |
| **What happens** | Input may be accepted by the TUI but handled differently than expected. |
| **Result** | **Unpredictable behavior.** |
| **Probability** | **LOW** |

### 7. Session Watcher Fails to Detect Turn Completion

| Attribute | Value |
|-----------|-------|
| **Terminal status** | Stuck on `"active"` → eventually `"waiting"` (15s timer fires) |
| **What could go wrong** | Several scenarios where turn completion is missed: |

**7a. JSONL file not found**
- `resolveSessionFile` returns null → watcher never starts.
- Terminal status follows the basic "active"→"waiting" cycle only.
- Probability: **LOW** after initial setup.

**7b. JSONL format changes**
- `checkTurnComplete` looks for specific JSON shapes (`stop_reason: "end_turn"`, `subtype: "turn_duration"`).
- If Claude Code changes its JSONL format, completion detection breaks.
- Terminal never transitions to "completed" — falls back to "waiting" after 15s.
- Impact: User can still send (both "active" and "waiting" are allowed), but lacks the semantic accuracy of "completed" status.

**7c. File system watcher misses events**
- macOS FSEvents can coalesce or drop events under heavy I/O.
- The debounce timer (300ms) provides some resilience but doesn't guarantee eventual delivery.
- Probability: **VERY LOW**

## Fundamental Design Gap

**There is no `"busy"` or `"generating"` status in the system.** The `TerminalStatus` type is:

```typescript
type TerminalStatus = "running" | "active" | "waiting" | "completed" | "success" | "error" | "idle";
```

The state machine has two problems:

1. **`"active"` conflates two opposite conditions:** "Claude is generating output" (not accepting input) and "Claude just produced output and may now be ready" (accepting input). These are semantically opposite but produce the same terminal status.

2. **No back-channel from Claude Code:** The only signals TermCanvas receives from Claude Code are:
   - PTY output bytes (used for active/waiting detection)
   - JSONL file changes (used for turn-completion detection)

   Neither signal provides real-time "Claude is ready for input" information. The JSONL completion signal is the closest proxy, but it has a 300ms debounce delay and only detects `end_turn` / `turn_duration`.

3. **The composer submit is fire-and-forget:** `writeToPty` writes bytes to the PTY and returns success regardless of whether the application consumed them. There is no delivery confirmation.

## Unresolved Problems

1. The fundamental issue — no way to distinguish "Claude is outputting" from "Claude is ready for input" using only PTY output activity — cannot be solved without either:
   - A protocol-level signal from Claude Code (e.g., a special escape sequence or API endpoint indicating readiness)
   - Content-based heuristic analysis of PTY output to detect prompt patterns
   - The existing session watcher JSONL approach, but used as a **gate** (blocking send until "completed") rather than just a status indicator

2. The permission prompt scenario (Case 2) is a safety concern. Composer should ideally detect permission prompts and either block sending or warn the user.

3. There is no queuing mechanism — if a message can't be delivered, it should be queued and sent when Claude becomes ready, rather than being silently dropped.
