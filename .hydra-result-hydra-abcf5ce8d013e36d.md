# TermCanvas 交互逻辑缺陷分析与本质解决方案

## 研究方法

通过并行分析以下5个维度，对 TermCanvas 现有交互架构进行深度审查：
1. Composer 交互机制（ComposerBar.tsx, composerTarget.ts, composerInputBehavior.ts, composerStore.ts）
2. 快捷键系统（shortcutStore.ts, useKeyboardShortcuts.ts, shortcutTarget.ts）
3. 焦点管理和输入路由（TerminalTile.tsx, cliConfig.ts, projectStore.ts, projectFocus.ts）
4. 剪贴板提交架构（electron/composer-submit.ts）
5. 替代架构方案（Warp, Zellij, Rio 参考）

---

## 一、焦点管理缺陷

### 1.1 核心问题：三层焦点系统不同步

TermCanvas 存在三层独立的焦点状态，它们之间缺乏原子性同步：

| 焦点层 | 存储位置 | 更新时机 | 同步性 |
|--------|---------|---------|--------|
| **逻辑焦点** | `projectStore.terminal.focused` | `setFocusedTerminal()` 同步更新 | 即时 |
| **Composer 焦点** | DOM `textarea.focus()` | `useEffect` 异步触发 (ComposerBar.tsx:160-164) | 延迟一帧 |
| **xterm DOM 焦点** | `xtermRef.current.focus()` | 仅点击 xterm 容器时 (TerminalTile.tsx:642-647) | 独立事件 |

**结果**：在逻辑焦点已更新但 Composer/xterm DOM 焦点尚未更新的时间窗口内，用户按键可能路由到错误的终端。

```
Frame 1 (同步): Store 更新 → React re-render → DOM 更新（边框颜色）
                ✗ Composer useEffect 尚未执行
                ✗ xterm.focus() 未被调用

Frame 2 (异步): Composer textarea.focus() 执行

如果用户在 Frame 1 和 Frame 2 之间按键 → 按键路由到旧焦点终端
```

### 1.2 焦点竞争抖动

**场景：快捷键切换终端** (useKeyboardShortcuts.ts:274-285)

```
用户按 Cmd+] 切换到下一个终端
  → setFocusedTerminal(nextTerminal.id)     // 逻辑焦点更新
  → targetTerminal 改变
  → ComposerBar useEffect 触发              // textarea.focus()
  → Composer 获得焦点（即使用户想要 xterm 焦点）
```

对于 `inputMode: "type"` 的终端（shell/lazygit/tmux），用快捷键切换后 xterm 不会获得 DOM 焦点（TerminalTile.tsx:643-647 只在 `onClick` 时触发），用户必须再点击一次 xterm 区域才能输入命令。

**场景：ComposerBar 自动对焦** (ComposerBar.tsx:160-164)

```typescript
useEffect(() => {
  if (targetTerminalId && isTargetReady) {
    textareaRef.current?.focus();  // 无条件对焦，无 debounce
  }
}, [targetTerminalId, isTargetReady]);
```

每当焦点终端改变时，无条件对焦 Composer textarea，不检查用户当前意图。

### 1.3 焦点丢失无自动恢复

- `normalizeProjectsFocus()` (projectFocus.ts:9-47) **仅在 `setProjects()` 时调用** (projectStore.ts:572)，运行时不会自动调用
- 删除焦点终端 (projectStore.ts:401-422) 时**不检查该终端是否是当前焦点**，焦点标记随终端消失
- 清除焦点 (projectStore.ts:559-570) 后**不提供新的焦点目标**，依赖用户手动恢复
- `setFocusedWorktree()` (projectStore.ts:546-557) 会**清除所有终端焦点但不指定新焦点**

### 1.4 焦点与 Composer 目标的概念混淆

`terminal.focused` 同时用于两个不同目的：
1. **输入路由** — 决定 xterm 是否获得 DOM 焦点
2. **Composer 目标** — `resolveComposerTarget()` (composerTarget.ts:50-54) 直接取 `focused === true` 的终端

这意味着用户无法在不聚焦某个终端的情况下向它发送 Composer 内容。

## 二、输入路由问题

### 2.1 剪贴板污染

这是当前架构最严重的用户体验问题。

**污染时间窗口**：

```
T=0ms   快照用户剪贴板 (snapshotClipboard)
T=50ms  写入图片到剪贴板 (writeClipboardImage)
T=170ms 写入第二张图片
T=290ms 写入文本到剪贴板 (writeClipboardText)
T=410ms 发送 Enter 键
T=411ms finally: 恢复快照 (restoreClipboard)

时间窗口内（0-411ms），用户的任何 Cmd+C 操作都会被后续覆盖
恢复的快照可能已过期（用户在 T=100ms 复制了新内容，但恢复的是 T=0ms 的内容）
```

**代码证据** (composer-submit.ts:299-381)：
- 快照在提交开始时抓取 (第299行)
- 恢复在 `finally` 块中执行 (第367-381行)
- 恢复失败仅转为 warning，不重试 (第369-380行)
- 没有日志或可观测性，用户无法知道剪贴板被污染

### 2.2 快捷键透传/拦截边界模糊

**三套独立的键盘处理系统**：

| 系统 | 作用域 | 文件位置 |
|------|--------|---------|
| 全局快捷键 | `window` 事件 | useKeyboardShortcuts.ts:405 |
| Composer 局部 | textarea `onKeyDown` | ComposerBar.tsx:381-395 |
| xterm 自定义 | xterm `attachCustomKeyEventHandler` | TerminalTile.tsx:173-178 |

**关键冲突**：`shouldIgnoreShortcutTarget()` (shortcutTarget.ts:20-24) 的逻辑是：
- 仅当在 editable 元素中 **且没有修饰符** 时才忽略快捷键
- 带修饰符的快捷键（Mod+T, Mod+[等）**永远不会被 textarea 拦截**
- 用户在 Composer 中输入时按 Mod+T 会意外新建终端

**两套 `hasPrimaryModifier()` 实现** (shortcutTarget.ts:13-18 和 shortcutStore.ts:46-51) 违反 DRY 原则，如果平台检测不同会导致行为不一致。

### 2.3 Draft 为空时的行为不一致

Arrow keys 转发使用 `draft.length === 0` (ComposerBar.tsx:115)，但提交检查使用 `draft.trim().length === 0` (ComposerBar.tsx:245)。

两个"空"的定义不一致：`draft = "   "` 时，arrow keys 不转发（length=3），但提交会报空错误（trim 后 length=0）。

### 2.4 Enter 提交 vs 换行判断

- 输入法兼容性只检查 `event.isComposing || event.nativeEvent?.isComposing` (composerInputBehavior.ts:10-22)
- 某些中文输入法可能在其他位置设置标志，导致误提交
- 没有 Ctrl+Enter 选项，某些用户习惯的提交方式不被支持

## 三、交互一致性问题

### 3.1 type vs paste 模式的体验差异

| 维度 | type 模式 (shell) | paste 模式 (claude) |
|------|-------------------|---------------------|
| 输入入口 | xterm 直接 | Composer textarea |
| 焦点行为 | 点击 xterm 获得焦点 | 点击 xterm 不获焦 |
| 图片支持 | ✗ | ✓ |
| 提交方式 | PTY 直接写入 | 剪贴板中介 |
| 延迟 | 0ms | 120ms+ |
| 剪贴板影响 | 无 | 被污染 |

**关键不一致**：点击同一个终端的不同区域导致焦点策略不同：
- 点击 tile header → `setFocusedTerminal()` → Composer 对焦
- 点击 xterm 区域 → 仅 type 模式时 xterm 对焦

### 3.2 自动类型检测的隐性影响

TerminalTile.tsx:282-316 中的自动检测可能在 3 秒后将终端从 "shell"（type）升级为 "claude"（paste），导致焦点行为**无警告地瞬间改变**。用户可能正在 xterm 中输入，突然发现键盘输入被 Composer 拦截。

### 3.3 Composer 不可用状态处理

- `targetState` 的 "empty"（无支持终端）和 "no-target"（有支持但无焦点）对用户来说差别不大，但错误提示完全不同 (composerTarget.ts:56-69)
- `disabled` 状态导致焦点丢失，`requestAnimationFrame` 恢复焦点 (ComposerBar.tsx:279) 有时间窗口
- 没有实时反馈终端状态变化：用户可能输入一大段文本后才发现无法提交

## 四、架构层面根本问题

### 4.1 剪贴板粘贴是否是正确架构？

**剪贴板粘贴的本质问题**：

1. **全局资源冲突**：剪贴板是操作系统级全局资源，无法对特定应用隐藏或上锁
2. **快照-操作-恢复模式的固有缺陷**：快照时刻 T0 与恢复时刻 Tn 之间可能相差数百毫秒，期间用户可干预
3. **图片传输的瓶颈**：PTY 是纯文本协议，无法直接传输二进制数据，剪贴板是唯一的系统级二进制通道
4. **平台差异**：粘贴快捷键因平台而异（Claude macOS 用 ESC+v, 其他用 Ctrl+V），不可控

**为什么仍然在用**：
- AI 终端（Claude/Codex）通过粘贴事件识别"用户想粘贴内容"的意图
- PTY 直接写入无法区分"用户输入"和"程序注入"
- 图片传输目前没有绕过剪贴板的标准方案

### 4.2 xterm.js 的限制

xterm.js 作为浏览器端终端模拟器：
- **DOM 焦点管理是浏览器原生的**，无法精确控制焦点转移时序
- **键盘事件通过浏览器事件冒泡**，与全局快捷键处理器存在竞争
- **没有提供"向终端发送输入但不获焦"的 API**，焦点和输入绑定在一起
- **渲染性能受限于 Canvas/WebGL**，无法与 Warp 的原生 GPU 渲染竞争

### 4.3 Electron IPC 复杂度

提交流程需要跨越三个进程边界：

```
React 渲染进程 → IPC → Electron 主进程 → PTY Manager → shell 进程
```

- `invoke` 没有显式超时机制，多张图片+延迟可能导致数秒等待
- 前端持有的 terminal 状态可能已过期，存在 TOCTOU 漏洞 (main.ts:491-498)
- 没有互斥锁防止并发提交竞争剪贴板

## 五、本质解决方案探索

### 5.1 方案对比：四种输入注入架构

| 方案 | 实现复杂度 | 剪贴板影响 | 图片支持 | 兼容性 | 延迟 |
|------|-----------|-----------|---------|--------|------|
| **A. PTY 直接写入** | 低 | 无 | ✗ | 高 | 极低 |
| **B. 剪贴板粘贴**（当前） | 中 | 严重 | ✓ | 中 | 120ms+ |
| **C. xterm.js paste API** | 低 | 无 | ✗ | 中 | 低 |
| **D. 自定义终端渲染** | 极高 | 无 | ✓ | 需自建 | 极低 |

### 5.2 参考：业界终端架构选择

**Warp Terminal**：
- 完全自定义 Rust + Metal/Vulkan 渲染，不用 xterm.js
- 直接控制 PTY，输入通过自定义输入层路由
- 能精确控制焦点、输入、渲染的每个环节
- 代价：海量工程投入，需要从零构建终端模拟

**Zellij**：
- Client-Server 架构，Server 管理所有 PTY
- 输入通过 InputHandler → Route Thread → Server → PTY 的清晰链路
- 每个子系统运行在独立线程，通过 message passing 通信
- PTY Writer Thread 专门负责异步写入，管理 resize 和缓存
- **启示**：输入路由应该有明确的分层和优先级，而非多个 event listener 竞争

**Rio Terminal**：
- Rust + WebGPU + Sugarloaf 自定义渲染器
- 复用 Alacritty 的 ANSI parser、事件系统和 grid 系统
- 纯原生架构，不受浏览器 DOM 焦点模型限制

### 5.3 推荐方案：分阶段演进

#### 阶段一：消除剪贴板污染（短期，低风险）

**方案 A+B 混合：文本用 PTY 直接写入，图片用临时文件**

```
文本提交：
  pty.write(text + "\r")  // 直接 PTY 写入，无需剪贴板

图片提交：
  1. 将图片写入临时文件 /tmp/termcanvas-img-{hash}.png
  2. 通过 PTY 写入文件路径（或使用 AI CLI 的文件引用语法）
  3. 不接触剪贴板
```

**前提条件**：需要确认 AI CLI（Claude/Codex）是否支持直接 PTY 文本输入（而非只监听粘贴事件）。如果 AI CLI 的 input handler 不区分粘贴和手动输入，这个方案可以直接工作。

**对于确实需要粘贴事件的 CLI**：
- 引入互斥锁防止并发提交
- 快照恢复前检测剪贴板是否在中间被修改（hash 比较）
- 如果被修改，不恢复旧快照（保留用户最新内容）
- 添加可观测性日志

#### 阶段二：统一焦点模型（中期）

**核心改造**：将三层焦点合并为单一焦点管理器

```typescript
interface FocusManager {
  // 原子性地设置焦点，同时更新逻辑焦点、Composer焦点、xterm DOM焦点
  setFocus(terminalId: string, source: "click" | "shortcut" | "auto"): void;

  // 焦点丢失时的自动恢复策略
  onFocusLost(terminalId: string): void;  // 自动转移到相邻终端

  // 焦点有效性验证
  validateFocus(): boolean;  // 检查焦点终端是否仍然存在且可用
}
```

**关键改进**：
- 快捷键切换焦点时，`type` 模式终端同时获得 xterm DOM 焦点
- 删除终端时自动转移焦点到相邻终端
- 终端类型变更时通知用户焦点行为将改变

#### 阶段三：输入路由分层（中期）

**参考 Zellij 的分层设计**：

```
Layer 1: 全局快捷键拦截（最高优先级）
  ↓ 未匹配
Layer 2: Composer 快捷键处理
  ↓ 未匹配
Layer 3: 终端输入路由（按 inputMode 分流）
  ├─ type 模式 → xterm 直接处理
  └─ paste 模式 → Composer textarea 处理
```

当前问题是三个层同时监听、互相竞争，没有明确的优先级链。

#### 阶段四：自定义终端渲染（长期，高投入）

如果 xterm.js 的限制成为瓶颈，考虑：
- 基于 Canvas/WebGL 的自定义终端渲染器（不依赖 xterm.js 的 DOM 焦点模型）
- 直接控制 PTY 读写，不经过 xterm.js 的事件系统
- 参考 Warp/Rio 的 GPU 渲染方案

**代价极高**，仅在 xterm.js 无法满足需求时考虑。

## 六、缺陷优先级排序

| 优先级 | 缺陷 | 影响 | 建议方案 |
|--------|------|------|---------|
| **P0** | 剪贴板污染 | 用户数据丢失 | PTY 直接写入 + 图片临时文件 |
| **P0** | 快捷键切换焦点后 xterm 不获焦 | type 模式终端无法键盘输入 | 焦点管理器统一更新 |
| **P1** | 焦点三层不同步 | 按键路由到错误终端 | 原子性焦点更新 |
| **P1** | 自动类型检测无警告改变焦点行为 | 用户输入被意外拦截 | 类型变更通知 |
| **P1** | 删除焦点终端后焦点丢失 | Composer 不可用 | 自动焦点转移 |
| **P2** | 快捷键透传/拦截边界模糊 | 意外触发全局快捷键 | 输入路由分层 |
| **P2** | Draft 空判断不一致 | 边界条件 bug | 统一空值定义 |
| **P2** | Composer disabled 焦点丢失 | 提交后焦点闪烁 | requestAnimationFrame → flushSync |
| **P3** | targetState 三态语义混淆 | 错误消息误导 | 简化状态模型 |
| **P3** | 两套 hasPrimaryModifier 实现 | 潜在不一致 | 抽取公共函数 |

## 七、结论

TermCanvas 的交互缺陷本质上源于**三个架构决策的连锁反应**：

1. **使用剪贴板作为输入中介** → 剪贴板污染、平台差异、时序问题
2. **逻辑焦点与 DOM 焦点分离** → 焦点竞争、抖动、不同步
3. **多个独立的键盘事件处理器** → 透传/拦截边界模糊、优先级不明确

短期应优先解决剪贴板污染（P0）和焦点同步（P0/P1），因为这些直接影响用户数据安全和基本可用性。中期应建立统一的焦点管理器和分层输入路由。长期是否需要自定义终端渲染取决于 xterm.js 的限制是否成为产品瓶颈。
