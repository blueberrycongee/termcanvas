## Status State Machine

```
                         ┌─────────────────────────────────────┐
                         │                                     │
    Terminal created ──► idle ──► (PTY spawned) ──► running    │
                                                      │        │
                                          PTY output  │        │
                                                      ▼        │
                                                   active      │
                                                      │        │
                                      15s no output   │        │
                                                      ▼        │
                                                   waiting     │
                                                      │        │
                              Session watcher detects  │        │
                              turn completion (JSONL)  │        │
                                                      ▼        │
                                                  completed    │
                                                      │        │
                                          PTY output   │ (loops back)
                                                      ▼        │
                                                   active ─────┘

                         Process exit code 0 ──► success
                         Process exit code ≠0 ──► error
```

### Key Status Lists

- **INTERACTIVE_STATUSES** (Composer `allowedStatuses` for ALL terminal types): `['running', 'active', 'waiting', 'completed', 'success', 'idle']`
- **READY_STATUSES** (defined but NOT directly used by ComposerBar): `['idle', 'waiting', 'completed', 'success']`

### Composer Gating (Two Layers)

1. **Textarea disabled** when `targetState !== "ready"` — but `getComposerTargetState()` returns `"ready"` simply if a focused terminal exists (does NOT check status at all!) — see `composerTarget.ts:56-69`
2. **handleSubmit() status check** at `ComposerBar.tsx:228`: `adapter.allowedStatuses.includes(targetTerminal.status)` — blocks `'error'` only (the only status NOT in INTERACTIVE_STATUSES)

**Critical finding:** The textarea is enabled for ALL statuses as long as a focused terminal exists. The submit handler blocks only `'error'`. This means the user can type and submit when the terminal is in ANY status except `'error'`.

## Identified Mismatches: Terminal Status vs Claude Code's Actual Input State

### Mismatch 1: `active` allows sending while Claude Code is generating (CRITICAL)

**Terminal status:** `active` (PTY producing output)
**Claude Code reality:** Generating a response — NOT reading stdin
**Composer behavior:** Allows submission (active ∈ INTERACTIVE_STATUSES)
**What happens:**
- `composer-submit.ts` writes text to PTY via `ptyManager.write()` then writes `\r` (Enter)
- For Claude Code (`inputMode: "paste"`): clipboard is loaded with text, paste key sequence (`ESC v` on macOS) is sent to PTY, then `\r`
- Claude Code's TUI is in output mode — **it does NOT have an input field active**
- The pasted text and Enter key are consumed by the TUI rendering layer
- **Result: `ok: true` returned to Composer, but the message is silently dropped by the terminal**

### Mismatch 2: `running` allows sending before Claude Code is ready

**Terminal status:** `running` (just spawned)
**Claude Code reality:** Still initializing — loading config, connecting to API, displaying splash
**Composer behavior:** Allows submission
**What happens:**
- Text/paste written to PTY arrives before Claude Code's input handler is initialized
- If Claude Code hasn't rendered its input prompt yet, the pasted content goes nowhere
- **Result: `ok: true`, message lost**

### Mismatch 3: `waiting` doesn't distinguish "waiting for user input" from "briefly paused during generation"

**Terminal status:** `waiting` (15s no output)
**Claude Code reality could be:**
- (a) Genuinely waiting for user input at the `>` prompt ✓
- (b) Paused during a long tool execution (e.g., running a test suite) — not accepting input ✗
- (c) Showing a permission prompt (Y/n) — will interpret ANY text as a response to the prompt ✗

**For case (c):** If user sends "refactor the auth module" via Composer while Claude Code shows `Allow tool use? (Y/n)`, the first character `r` gets interpreted as permission denial (anything except Y/y is treated as rejection by Claude Code).

### Mismatch 4: Transition from `completed` back to `active` has no "accepting input" intermediate state

**Terminal status:** `completed` (session watcher detected end_turn)
**Claude Code reality:** Has finished generating, likely showing the `>` prompt
**Composer behavior:** Allows submission ✓
**Race condition:** After the turn completes, Claude Code writes its final output (the prompt `>`). This output triggers `active` status. Between the session watcher detecting completion and the final output arriving, there's a brief window where status is `completed` but Claude Code is still rendering. However, this is actually SAFE — Claude Code's input is ready when the turn completes, and any delay in rendering doesn't affect stdin acceptance.

## Race Conditions in Status Transitions

### Race 1: Session Watcher Debounce Delay (300ms)

`session-watcher.ts:182-198`: The watcher debounces JSONL file changes by 300ms. After Claude Code finishes a turn:
1. Claude Code writes `end_turn` to JSONL → watcher debounce timer starts (300ms)
2. Claude Code continues writing final output (prompt, etc.) → resets debounce timer
3. After 300ms of no more JSONL writes → `checkTurnComplete()` runs → fires callback
4. IPC sends `session:turn-complete` to renderer
5. `TerminalTile.tsx` updates status to `completed`

**Gap:** ~300-500ms between Claude Code actually finishing and TermCanvas knowing it. During this gap, status is `active` or `waiting`, and Composer allows sending. This is the **same "Mismatch 1" window** but from the timing perspective.

### Race 2: Session ID Polling Delay (up to 30 seconds)

`pollSessionId()` polls every 2 seconds, up to 15 attempts. Until a session ID is found:
- No session watcher is active
- Status transitions rely solely on PTY output activity
- `completed` status is NEVER reached (requires session watcher)
- Status oscillates between `active` and `waiting` based on output alone

**Impact:** During the first ~4-30 seconds after launch, there is no turn-completion awareness at all.

### Race 3: No Lock Between Status Check and PTY Write

`ComposerBar.tsx:228` checks `adapter.allowedStatuses.includes(targetTerminal.status)` at submit time. But between this check and the actual PTY write in `composer-submit.ts`, Claude Code's state could change (e.g., it starts generating in response to a permission grant). There is no mutex — the status check is advisory only.

## Specific Scenarios: Composer Returns `ok: true` But Message Is Lost

### Scenario 1: Send During Active Generation
1. Claude Code is generating a long response (outputting text continuously)
2. Terminal status: `active`
3. User types message in Composer and hits Enter
4. `handleSubmit()` → status check passes (`active` ∈ INTERACTIVE_STATUSES)
5. `composer-submit.ts` → clipboard set to message text → paste key (`ESC v`) sent to PTY → `\r` sent
6. Claude Code TUI receives paste sequence while in output mode — **text is lost**
7. Returns `{ ok: true }`

### Scenario 2: Send During Permission Prompt
1. Claude Code shows `Allow tool use? (Y/n)`
2. Terminal status: `waiting` (no output for 15s while waiting for user)
3. User sends "please also add tests" via Composer
4. `composer-submit.ts` pastes text, sends `\r`
5. Claude Code interprets first character as permission response → rejects tool → tries to process remaining text as new prompt
6. **Permission unintentionally denied, next prompt is garbled**
7. Returns `{ ok: true }`

### Scenario 3: Send Before CLI Ready
1. Claude Code just launched, showing initialization output
2. Terminal status: `running` then quickly `active`
3. User sends message immediately
4. Claude Code hasn't rendered its input prompt yet
5. Paste sequence sent but no paste handler active in Claude Code
6. **Text lost**
7. Returns `{ ok: true }`

### Scenario 4: Send During Tool Execution
1. Claude Code is running a long tool (e.g., `npm test`)
2. No output for 15s → status becomes `waiting`
3. User sends new message via Composer
4. Claude Code is not at its input prompt — it's waiting for tool completion
5. Paste arrives in PTY but Claude Code's TUI ignores it
6. **Text lost**
7. Returns `{ ok: true }`

### Scenario 5: Send During Spinner
1. Claude Code shows thinking spinner (e.g., `⠋ Thinking...`)
2. Spinner updates produce output → status `active`
3. User sends message
4. Same as Scenario 1 — Claude Code is in output mode
5. **Text lost**
6. Returns `{ ok: true }`

## Root Cause Summary

The fundamental problem is that **TermCanvas tracks PTY output activity, not Claude Code's input readiness**. The status model has no concept of "the CLI is ready to accept a new prompt." The only reliable signal for this is `completed` (from the session watcher), but:

1. `completed` is not the only status that allows submission — `active`, `running`, and `waiting` also allow it
2. There is no "busy/generating" status that would block the Composer
3. The session watcher only fires at turn completion, not at the start of input readiness
4. The Composer returns `ok: true` based on whether the PTY write succeeded, not whether Claude Code actually processed the input

### Potential Solution Directions (not implemented — read-only investigation)

1. **Add a `busy` / `generating` status** that blocks the Composer when output is flowing (status = `active`) for AI CLI terminals specifically
2. **Only allow sending for AI CLIs in `completed` or `waiting` status**, with `waiting` requiring a heuristic check that the CLI is at its input prompt (not in a tool execution pause)
3. **Use the session watcher bidirectionally**: detect both turn-completion AND turn-start, to toggle between `completed` (sendable) and `generating` (blocked)
4. **Buffer & retry**: If the message cannot be delivered, hold it and retry when the CLI becomes ready (but this adds complexity and doesn't solve the permission prompt issue)

## Unresolved Problems
- No reliable way to distinguish "waiting for user input at prompt" from "waiting during tool execution" using PTY output alone
- Permission prompts (Y/n) are indistinguishable from the regular input prompt at the status-detection level
- The `ok: true` return from composer-submit is fundamentally unreliable for TUI-based CLIs — it only confirms the PTY write succeeded, not that the application consumed the input
