Issues found

1. Blocking: the re-entrancy guard does not actually guarantee single-submit semantics.
   - In [src/components/ComposerBar.tsx](/Users/zzzz/termcanvas/src/components/ComposerBar.tsx#L214), `handleSubmit` reads `isSubmitting` from the render-time closure. The first invocation only flips the store at [src/components/ComposerBar.tsx](/Users/zzzz/termcanvas/src/components/ComposerBar.tsx#L262), so a second rapid Enter/click can still invoke the same stale callback instance before React re-renders with `isSubmitting=true`.
   - That means commit `b575c3a` does not fully fix the race described in its message. A ref or a store getter read at call time is needed for a real re-entrancy latch.

2. Blocking: `READY_STATUSES` is too coarse and can block legitimate sends after launch.
   - Claude/Codex now only allow `idle|waiting|completed|success` in [src/terminal/cliConfig.ts](/Users/zzzz/termcanvas/src/terminal/cliConfig.ts#L35) and [src/terminal/cliConfig.ts](/Users/zzzz/termcanvas/src/terminal/cliConfig.ts#L82).
   - But terminal readiness is tracked via output activity, not prompt detection. New PTYs are set to `running` at [src/terminal/TerminalTile.tsx](/Users/zzzz/termcanvas/src/terminal/TerminalTile.tsx#L221), then any output flips them to `active` at [src/terminal/TerminalTile.tsx](/Users/zzzz/termcanvas/src/terminal/TerminalTile.tsx#L353), and they do not become `waiting` until 15s of silence at [src/terminal/TerminalTile.tsx](/Users/zzzz/termcanvas/src/terminal/TerminalTile.tsx#L358).
   - Result: a freshly launched Claude/Codex terminal can sit at a visible prompt while Composer still rejects submission because status remains `active`. This fixes silent loss during generation, but it also introduces false negatives during normal prompt-ready states.

3. Blocking: the diff leaves the relevant adapter test suite broken.
   - `tests/terminal-adapters.test.ts` still asserts old paste-mode behavior for Claude/Codex at [tests/terminal-adapters.test.ts](/Users/zzzz/termcanvas/tests/terminal-adapters.test.ts#L10) and [tests/terminal-adapters.test.ts](/Users/zzzz/termcanvas/tests/terminal-adapters.test.ts#L20).
   - `node --experimental-strip-types --test tests/terminal-adapters.test.ts` currently fails because the code now returns `bracketed-paste`, empty paste sequences, and different image fallback behavior.
   - Even if the implementation were correct, the branch is not in a passing state for the test script that already includes this file.

Non-blocking concerns

1. The status-gating commit does not actually “only enable sending”.
   - `getComposerTargetState()` ignores terminal status in [src/components/composerTarget.ts](/Users/zzzz/termcanvas/src/components/composerTarget.ts#L56), so the textarea/button stay enabled as long as there is a focused PTY. The actual block only happens inside `handleSubmit` at [src/components/ComposerBar.tsx](/Users/zzzz/termcanvas/src/components/ComposerBar.tsx#L230).
   - That is a UX mismatch with the commit message and will feel flaky to users.

2. Claude image behavior changed from “paste image” to “paste file path”, and the repo does not prove those are equivalent.
   - The local Claude keybinding research still maps clipboard image paste to `chat:imagePaste` at [docs/claude-code-keybindings.md](/Users/zzzz/termcanvas/docs/claude-code-keybindings.md#L112).
   - The new Claude test only proves that a staged file path is bracketed-pasted into the PTY at [tests/composer-submit.test.ts](/Users/zzzz/termcanvas/tests/composer-submit.test.ts#L189); it does not show that Claude interprets the pasted path as an attached image instead of plain text.
   - I would treat this as unverified behavior rather than a proven regression, but it needs manual validation or an integration test.

Suggestions
- Replace the `isSubmitting` closure check with a real in-flight latch (`useRef`, or a synchronous getter from the Zustand store) so duplicate invocations cannot pass through before re-render.
- Gate Claude/Codex sends on a stronger readiness signal than `TerminalStatus` activity buckets. At minimum, avoid blocking the initial prompt-ready window; ideally key off explicit prompt detection or a terminal-specific “input-ready” signal.
- Align the UI with the new gating rules so the textarea/button disable state matches the actual submit rules.
- Update [tests/terminal-adapters.test.ts](/Users/zzzz/termcanvas/tests/terminal-adapters.test.ts) for the new adapter contract and add coverage for the status-gating path in `ComposerBar`.
- Manually verify Claude image submission with staged-path bracketed paste before treating clipboard-image parity as solved.
- Leave `kimi`/`gemini`/`opencode` on clipboard paste for now unless they are known to support bracketed paste; they still have clipboard-race risk, but switching them without terminal-specific evidence would be speculative.
- `pasteKeySequence` for bracketed-paste terminals is effectively dead config today. It is harmless, but the adapter shape could be split by input mode later to make this explicit.

Overall assessment
- The bracketed-paste change likely improves text reliability for Claude and matches the existing Codex path.
- The series is not ready as-is. One fix is incomplete (`isSubmitting`), one fix is over-restrictive (`READY_STATUSES` vs. actual prompt readiness), and the committed test set is currently inconsistent with the implementation.
