# Hydra Orchestration Guide

## Architecture boundaries

- Hydra is a control plane, not an agent runtime. Agent execution is delegated to local `claude` / `codex` CLIs through `termcanvas terminal create --prompt`.
- `termcanvas terminal input` is not a supported task-dispatch path for Claude/Codex automation. Launch a fresh agent terminal with `termcanvas terminal create --prompt` instead.
- File evidence is authoritative. `workflow.json`, `assignment.json`, `task.md`, `artifacts/*.md`, and `result.json` define the runtime. Terminal prose is not a source of truth.
- Internal orchestration state lives under repo-local `.hydra/workflows/<workflow>/`.
- Workflow progression is state-machine-driven. `pending -> claimed -> in_progress -> completed|timed_out|failed` is explicit and idempotent at the assignment level.
- Completion is strict-fail. Missing or invalid `result.json` keeps the active run waiting or fails it; Hydra does not infer success from terminal output.
- Current workflow templates are staged, not fan-out parallel graphs. `hydra spawn` remains a direct isolated-worker primitive when the split is already known.

## Mode selection

- Work directly in the current agent when the task is simple, local, or faster than paying workflow overhead.
- Use `hydra run --template single-step` when you want one implementer plus worktree isolation, file evidence, and retry/status control.
- Use default `hydra run` for the `researcher -> implementer -> tester` loop on ambiguous, risky, or PRD-driven tasks.
- Use `hydra spawn` when you already know the split and only need one isolated worker terminal, not a full workflow run.

## Runtime SOP

1. Start the chosen path:
   ```bash
   hydra run --task "..." --repo .
   ```
   The default workflow is `researcher -> implementer -> tester`, with approval after each successful research pass. For a single direct implementer run, pass `--template single-step`. For a direct isolated worker primitive, use `hydra spawn --task "..." --repo .`.
2. Advance a workflow run once:
   ```bash
   hydra tick --repo . --workflow <workflowId>
   ```
3. Watch until terminal state:
   ```bash
   hydra watch --repo . --workflow <workflowId>
   ```
4. Inspect structured status and failure reasons:
   ```bash
   hydra status --repo . --workflow <workflowId>
   ```
5. Retry when the workflow is retryable:
   ```bash
   hydra retry --repo . --workflow <workflowId>
   ```
6. Inspect or clean up direct workers:
   ```bash
   hydra list --repo .
   hydra cleanup <agentId> --force
   ```
7. Clean up workflow runtime state and worktrees:
   ```bash
   hydra cleanup --workflow <workflowId> --repo . --force
   ```

## Runtime files

```
.hydra/workflows/<workflowId>/
  workflow.json
  inputs/
    user-request.md
    revision-request.md
  assignments/
    <assignmentId>/
      assignment.json
      runs/
        <runId>/
          task.md
          result.json
          artifacts/
            brief.md
            approval-request.md
```

- `workflow.json`: workflow metadata, status, current assignment, approved refs.
- `assignment.json`: assignment metadata, role, source assignment, active run, retry history.
- `task.md`: thin task sheet generated by Hydra for the current run.
- `artifacts/*.md`: human-readable deliverables written by the agent.
- `result.json`: machine-readable completion result for the active run. This is the only completion gate.

## Result contract

`result.json` must include:

- `schema_version`
- `workflow_id`
- `assignment_id`
- `run_id`
- `success`
- `summary`
- `outputs[]`
- `evidence[]`
- `next_action`

Hydra rejects:

- missing `result.json`
- malformed JSON
- schema mismatch
- wrong `workflow_id`
- wrong `assignment_id`
- wrong `run_id`
- invalid `next_action`

## Troubleshooting

- Workflow stuck in `running`:
  - Run `hydra status --repo . --workflow <workflowId>`
  - Check the active assignment and run
  - If `result.json` is missing, the run is still active, stalled, or awaiting contract publication
- Workflow failed with `COLLECTOR_RESULT_INVALID`:
  - Open the active run's `result.json`
  - Fix malformed JSON or missing required fields
- Workflow failed with timeout:
  - Inspect the active assignment and retry budget via `hydra status`
  - Use `hydra retry` if the workflow is still retryable
- Cleanup refused because a workflow is still running:
  - Re-run with `--force` after confirming the active terminal can be destroyed safely

## Anti-patterns

- Treating terminal text as completion evidence
- Hacking tests or fixtures so the result looks green
- Adding silent fallbacks or swallowed errors just to keep the workflow moving
- Declaring success without a valid `result.json`
- Re-dispatching an assignment without idempotency checks

## Acceptance

Run the local acceptance harness:

```bash
cd hydra
npm run e2e:acceptance -- --repo /absolute/path/to/repo --report /absolute/path/to/report.md
```

The acceptance harness:

- launches real Claude/Codex terminals through TermCanvas
- drives a researcher -> implementer -> tester loop with approval and retry behavior
- verifies failure, retry, recovery, and cleanup
- writes a reproducible acceptance report
