Issues found
- Dirty tracking should not be implemented as a full serialized-snapshot comparison of the current save payload. `src/App.tsx` serializes volatile runtime fields such as terminal scrollback and session IDs (`src/App.tsx:55-82`), while terminal status/session data mutate continuously in the background (`src/terminal/TerminalTile.tsx:381-399`, `src/terminal/TerminalTile.tsx:500-597`). That would make the document become dirty again immediately after save during normal AI output. A better model is: keep a canonical saved-document hash for stable document data, but mark dirty from intentional document mutations only, excluding runtime-only recovery fields.
- Putting `workspacePath` and `dirty` into `canvasStore` is the wrong ownership boundary. `canvasStore` only owns viewport/panel UI state (`src/stores/canvasStore.ts:9-18`), while document mutations are spread across `projectStore`, `drawingStore`, and `browserCardStore` (`src/stores/projectStore.ts:23-109`, `src/stores/drawingStore.ts:57-69`, `src/stores/browserCardStore.ts:13-17`).
- A fixed 60-second autosave is too coarse for recovery and too blunt for performance. Recovery currently depends on serializing every terminal buffer (`src/App.tsx:55-82`, `src/terminal/terminalRegistry.ts:27-36`), so a blind timer risks both losing up to 60 seconds of work and doing expensive whole-canvas writes even when nothing changed. Prefer dirty-triggered debounced saves plus a longer max-interval backstop.
- The proposed graceful shutdown needs to cover every PTY teardown path, not just app close. Terminals are also destroyed when a tile unmounts or a workspace is replaced (`src/terminal/TerminalTile.tsx:600-623`, `src/components/Sidebar.tsx:106-113`, `src/App.tsx:152-165`). If only `app:close-confirmed` becomes graceful, switching/opening workspaces will still send abrupt hangups.
- A hard-coded 2-second `SIGTERM -> SIGKILL` window is risky for AI CLIs. The app currently spawns PTYs via `node-pty` (`electron/pty-manager.ts:13-37`) and kills them immediately (`electron/pty-manager.ts:119-131`, `electron/main.ts:695-703`). Some CLIs may need longer to flush session JSONL/index files, and depending on launch shape the signal may hit the shell/PTY leader rather than the actual child CLI. The timeout should be configurable and instrumented; 5-10 seconds with early exit on process termination is safer.
- The close/open flow still misses unsaved-change prompts on destructive non-close actions. Opening a workspace currently overwrites in-memory state immediately with no confirmation (`src/components/Sidebar.tsx:106-113`, `src/App.tsx:152-165`). “New workspace” or any future “clear canvas” action needs the same save/discard/cancel gate as window close.
- “Don’t Save” needs better semantics than simply clearing `state.json`. The current discard path writes an empty recovery file (`src/App.tsx:199-205`), which would also discard the last autosaved recovery state. With autosave enabled, discard should revert recovery state to the last clean baseline or suppress restoring this session, not blindly zero the file.
- Save failure handling should not force-close. The current close flow closes even when save throws (`src/App.tsx:193-196`). The proposal should keep the window open and surface retry/cancel on save errors.
- `state.json` writes should be atomic. `StatePersistence.save` currently writes directly to the target file (`electron/state-persistence.ts:25-27`), which risks a truncated recovery file on crash/power loss during autosave.
- Autosave should default to `state.json` only, not silently overwrite the workspace file. Unprompted writes to the user-chosen `.termcanvas` file change the semantics of Save, can surprise users, and may be problematic if that file is synced or versioned. If workspace-file autosave is desired, make it an explicit opt-in.
