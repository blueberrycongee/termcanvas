Most tools have some version of “save this for later”: bookmarks, favorites, todos, issues, chat history. They all appear to solve the same problem: put something down now so you can return to it later.

In AI coding workflows, that is not the hard part.

The hard part is whether the thing still has enough context when it comes back. A screenshot of a layout bug, a failing command, a product decision from a review, or a thread of work an agent left half-finished may be small on its own. Once it is scattered across terminal scrollback, chat logs, and browser tabs, it slowly loses its ability to be acted on.

A few days later, you may still remember that something mattered. You no longer know exactly what it was, why it mattered, or who should pick it up next.

That is the problem pins are meant to solve.

## The ceiling of interactive agents

The first way most people use a coding agent is as a smarter terminal session. Open a window, give it a task, wait for output, review the result, and steer the next turn.

That works well with one or two agents. It starts to break down when the workflow scales.

Once you have three to five sessions running, the attention cost becomes obvious. You have to remember what each agent is doing, which worktree it belongs to, whether it is stalled, whether its output can be trusted, and whether it discovered a follow-up that should not be lost.

The agents may be fast. Human attention becomes the bottleneck.

The awkward part is that many important discoveries are not ready to become formal tasks. An agent may notice a real issue while implementing something else. A reviewer may spot a small product problem that should not block the current change. A user may send a screenshot while you are focused on another branch.

These fragments are too concrete to keep in your head, but too early to promote into full project management.

So they usually fall into chat history.

Chat history is a timeline, not a workspace. It can reconstruct a conversation, but it is a poor handoff surface. A future agent, a collaborator, or even you next week should not have to read dozens of turns to guess what work is supposed to continue.

## The shift: not saving notes, preserving resumable work

A pin is not primarily a saved note.

It is a small piece of unfinished work fixed near the project.

That difference matters. A bookmark asks whether something is worth keeping. A pin asks whether future work can resume from it. A useful pin is not just a title; it carries enough context to restart the loop.

A weak pin says:

```text
Fix login
```

That tells you something exists. It does not tell you why, how to reproduce it, what good looks like, or who should handle it.

A better pin is a small handoff packet:

```text
Title: Fix the login button on narrow screens

Context: Reproduced in a 13-inch window. The primary button touches
helper text below it.

Evidence: Screenshot attached.

Expected: Keep stable spacing when the English copy wraps.

Next action: Ask an agent to locate the layout component and test the
narrow viewport.
```

The point is not to make note-taking heavier. The point is to make the future cheaper. The more complete the context, the lower the cost of restarting the work.

## Turning scattered project context into a control surface

Pins are useful because they live near the project.

External todo tools are often too far from the code. You can write down a task, but when it is time to act you still need to find the repo, branch, terminal, screenshot, and discussion that made the task meaningful. For normal work, that is friction. For agentic work, it directly lowers output quality, because thin context makes agents guess.

In TermCanvas, pins are not a global memo list. They are project working memory. They live beside repos, worktrees, terminals, and agent sessions. When you open a project, the pin list does not answer “what did I save?” It answers: what is still open here, what can be handed to an agent, what needs human judgment, what is done, and what was deliberately dropped?

That makes pins closer to a lightweight control surface.

When a pin is dropped onto a terminal or given to an agent, it stops being a note and becomes input. The title, body, links, screenshots, and next action become a compact context packet for the next worker.

That is the difference between a pin and a typical todo. A todo usually describes what should be done. A pin tries to preserve where to continue from.

## The workflow changes

When recording and handing off context becomes cheap, people start preserving things that previously were not worth the effort.

That does not necessarily mean doing more work. It means keeping more options alive at a lower cost.

A possible refactor can be pinned for later exploration. A UI bug can be captured with a screenshot without interrupting the current task. A review can produce a follow-up item instead of smuggling unrelated work into the current PR.

Pins give those fragments a place to go. They do not have to become formal issues immediately. They also do not have to pollute the current agent’s task boundary.

This matters even more when multiple agents are running. The danger is not that agents are slow. The danger is that their context gets mixed together. Pins provide a cheap routing layer: the current task continues, side discoveries are fixed in place, and the right human or agent can pick them up later.

## Tradeoffs and boundaries

More pins are not automatically better.

If everything becomes a pin, the list turns into another junk drawer. A healthy pin list is alive: open means it still needs attention, done means the loop is closed, and dropped means you deliberately decided not to continue.

Dropped matters. Some work is not worth doing, but it is still worth preserving the reason. Otherwise, the same idea will reappear weeks later and cost you another round of judgment.

Pins are also not a replacement for a project management system. Large features, cross-team plans, and scheduled work should still become issues or roadmap items. Pins are best for the space between “not now” and “do not lose this”: screenshots, traces, follow-ups, half-formed ideas, and small tasks an agent can pick up later.

## What pins are for

AI coding changes the developer’s role. You are no longer only writing code. You are managing agents, worktrees, and unfinished context across several threads of work.

In that workflow, the scarce resource is not another terminal window. It is resumable context.

Pins fix that context near the project so a future human or agent can continue from a clear handoff point instead of digging through chat history and terminal scrollback.

A pin is not a bookmark.

It is project working memory.
