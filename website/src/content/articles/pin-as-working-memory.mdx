很多工具里都有“收藏”“书签”“稍后读”。TermCanvas 里的 Pin 很容易被误解成同一类东西：看到一个想法，点一下，先放起来。

但如果只把 Pin 当成收藏夹，它的价值会被低估。

在 AI 编码里，真正麻烦的不是“我有没有记下这件事”，而是“当我回到这件事时，我还能不能恢复当时的上下文”。一个 bug 的截图、一段用户反馈、一个没有展开的产品想法、一个需要交给 agent 继续处理的问题，如果只留在聊天记录或终端历史里，过几天再看时往往只剩一个模糊的标题。

Pin 想解决的是这个断裂。

## AI 工作流里最容易丢的是半成品上下文

人类写代码时，经常会自然地暂停一件事：

- 这个按钮有点歪，先记一下。
- 这个错误状态以后要补。
- 这个 issue 需要等下个版本一起处理。
- 这个截图里的布局问题可以交给 agent 修。
- 这个想法现在不做，但不能忘。

传统待办工具能保存标题，却很难保存当时的工程现场。聊天工具能保存上下文，却会被新的对话冲走。GitHub issue 足够正式，但很多念头还没到需要建 issue 的程度。

AI 编码放大了这个问题。因为 agent 会不断产生中间状态：它跑过的命令、看到的错误、做了一半的判断、用户临时补充的截图。不是每件事都应该立刻执行，也不是每件事都值得变成正式 issue，但它们都可能在之后变得重要。

Pin 的位置就在这里：它是介于“脑子里记着”和“正式任务系统”之间的一层工作记忆。

## 一个好的 Pin 应该能让未来的人接手

Pin 不应该只写一句“修一下登录”。这样的记录过几天会变成谜语。

更有用的写法是把它当成一张很小的交接卡：

```text
标题：修复登录页按钮在窄屏下错位

背景：在 13 寸窗口宽度下复现，主按钮会贴到底部说明文字。

期望：按钮和说明之间保持稳定间距，英文文案变长时也不能重叠。

证据：附上截图。

下一步：让 agent 先定位布局组件，再补一个窄屏视觉检查。
```

这不是形式主义。它的目标很简单：未来的你、另一个协作者、或者一个新的 agent 读到它时，不需要重新追问“这是什么意思”。

Pin 支持正文、链接、图片和状态，原因也在这里。文字负责说明判断，链接负责连接外部来源，图片负责保留视觉证据，状态负责区分“还要做”“已经做完”“暂时放弃”。

## Pin 应该贴近项目，而不是漂在工具外面

很多待办工具的问题是离代码太远。你可以在外部工具里写下一条任务，但当你真正要处理它时，还要重新找到仓库、分支、终端、截图和相关上下文。

TermCanvas 把 Pin 放在项目旁边，是为了让它更接近实际发生问题的地方。

一个 Pin 不是抽象的提醒，它应该属于某个 repo、某个正在进行的项目、某片工作现场。这样它不会变成孤立的文字，而是能和终端、worktree、会话一起被管理。

当你打开一个项目时，Pin 列表应该回答的问题不是“我收藏过什么”，而是：

- 这个项目还有哪些未闭环的事情？
- 哪些想法已经有足够上下文，可以交给 agent？
- 哪些问题已经做完，可以从视野里收起来？
- 哪些被搁置的判断以后可能要重新打开？

这和普通 todo list 的差别在于，Pin 不是按人生或日程组织的，而是按工程现场组织的。

## 把 Pin 交给 agent 时，它变成上下文包

Pin 最有意思的地方，是它不只用于记录，还可以继续参与工作流。

当一个 Pin 被拖给终端或 agent，它表达的不是“请记住这个标题”，而是“这是当前对话应该继承的上下文”。标题、正文、截图和链接一起变成下一轮工作的输入。

这对 AI 编码很重要。因为 agent 的能力很大程度取决于上下文质量。你给它一句“修一下这个”，它会猜。你给它一张包含背景、证据、期望和下一步的 Pin，它就更接近一个明确的任务交接。

这也是 Pin 和聊天记录的差别。聊天记录是时间流，Pin 是被整理过的上下文包。时间流里什么都有，但不一定能复用；上下文包更小，却更适合被再次交给人或 agent。

## 怎么用 Pin 才不会变成垃圾箱

Pin 如果没有使用纪律，也会变成另一个堆积场。我的建议是把它限制在三类事情上。

第一类是**可执行但现在不做**的事情。比如 bug、明显的体验问题、可以交给 agent 的小任务。这类 Pin 要写清楚期望和下一步。

第二类是**需要证据保留**的事情。比如截图、报错、用户反馈、某个网页或 issue。没有证据，之后就很难判断当时为什么记录它。

第三类是**还没有成熟到 issue 的想法**。这类 Pin 可以更短，但也要写清楚为什么值得保留，以及什么时候应该重新看。

反过来，有些东西不适合 Pin。纯粹的长期愿望不适合，应该进产品路线图。已经确定要做的大任务不适合，应该进正式 issue 或项目管理系统。只是一句临时感叹也不适合，除非它背后有具体行动。

Pin 的价值不是无限保存，而是帮助你判断哪些上下文值得带到未来。

## 完成和搁置同样重要

Pin 有 open、done、dropped 这些状态，不只是为了列表好看。

完成代表闭环。搁置代表你有意识地决定暂时不做。两者都比“永远开着”健康。

AI 编码工具很容易制造一种错觉：只要事情被记录下来，就算被管理了。但真正的管理发生在后面：定期看一眼，完成就关掉，不值得做就搁置，信息不够就补上下文。

一个好的 Pin 列表应该是流动的。它不是仓库，也不是墓地，而是当前项目还需要注意的工作记忆。

## Pin 的本质

Pin 的本质不是收藏，而是把“未来需要恢复的上下文”固定下来。

它让一些还没有进入正式流程、但又不应该消失的东西，有一个靠近项目的位置。它让截图、链接、判断和下一步行动留在一起。它也让 agent 接手任务时，不必从一段散乱的聊天历史开始猜。

在单人写代码时，这只是减少遗忘。

在 AI agent 参与编码时，这会变成更重要的能力：让上下文可以被保存、整理、转交和重新启动。
