很多工具里都有“稍后处理”的入口：收藏、书签、待办、issue、聊天记录。它们看起来都能解决同一个问题：把现在不处理的事情放到未来。

但在 AI 编码工作流里，我们遇到的瓶颈不是“有没有地方记下来”。真正的瓶颈是：当这件事回到你面前时，它还剩下多少上下文。

一个布局问题的截图、一段失败日志、一次产品讨论里的判断、一个 agent 做到一半留下的线索，单独看都不大。可它们散落在终端历史、聊天记录和浏览器标签页里之后，就会慢慢失去可执行性。几天后你还记得“这里有个问题”，但不再确定问题是什么、为什么重要、下一步应该交给谁。

这就是 Pin 想解决的问题。

## 交互式 agent 的上限

刚开始使用编码 agent 时，我们很容易把它当成一个更聪明的终端会话。开一个窗口，给一个任务，等它跑完，读结果，再继续下一轮。

这个模式在一两个 agent 时很好用。问题出现在规模稍微变大之后。

当你同时开三到五个会话时，注意力成本会突然变得明显。你需要记住每个 agent 在做什么、它属于哪个 worktree、它有没有卡住、它刚才产出的结果该不该信、有没有后续动作应该被保存下来。agent 本身可能很快，但人的注意力开始成为瓶颈。

更麻烦的是，很多重要信息并不会自然变成正式任务。

agent 可能在实现过程中发现一个无关但真实的问题。你可能在 review 时看到一个值得之后处理的边角。用户可能发来一张截图，但你现在正在忙另一个分支。这些信息太具体，不能只靠脑子记；又太早，没必要立刻变成完整 issue。

于是它们通常会被丢进聊天记录里。

聊天记录的问题是，它是时间流，不是工作台。时间流很适合还原对话，却不适合承载未来任务。你不能指望一个新的 agent、未来的自己，或者另一个协作者从几十轮上下文里猜出“这里真正要继续的是什么”。

## 视角转换：不是记录想法，而是保留可恢复的工作

Pin 的设计出发点不是“收藏一个想法”。

更准确地说，Pin 是把一段尚未完成的工作固定在项目旁边。

这听起来只是措辞变化，但差异很大。收藏关心的是内容是否值得保存；Pin 关心的是未来能不能恢复执行。一个好的 Pin 不只是标题，它应该带着足够的信息让下一轮工作重新启动。

比如一句“修一下登录页”几乎没有价值。它只告诉你有事要做，却没有告诉你为什么、怎么复现、期望是什么、谁适合接手。

更好的 Pin 会像一个很小的交接包：

```text
Title: Fix the login button on narrow screens

Context: Reproduced in a 13-inch window. The primary button touches
helper text below it.

Evidence: Screenshot attached.

Expected: Keep stable spacing when the English copy wraps.

Next action: Ask an agent to locate the layout component and test the
narrow viewport.
```

这不是为了让记录变复杂，而是为了让未来变便宜。上下文越完整，重新启动这件事的成本越低。

## 把项目里的零散线索变成控制面

Pin 真正有用的地方，是它贴着项目存在。

外部待办工具通常离代码太远。你可以在那里写下任务，但真正处理时还要重新找到仓库、分支、终端、截图和讨论背景。对于普通任务，这只是麻烦；对于 AI agent 工作流，这会直接降低任务质量，因为 agent 接收到的上下文越薄，就越容易猜。

TermCanvas 里的 Pin 不是全局备忘录，而是项目里的工作记忆。它和 repo、worktree、terminal、agent 会话处在同一个工作现场里。打开项目时，你看到的不只是“我有哪些收藏”，而是这个项目还悬着哪些线索：哪些可以交给 agent，哪些需要人工判断，哪些已经完成，哪些应该搁置。

这让 Pin 更接近一个轻量控制面。

当一个 Pin 被拖给终端或 agent，它就不再只是记录。它变成输入。标题、正文、链接、截图和下一步一起组成一个上下文包，交给下一轮执行者。

这也是 Pin 和传统 todo 的关键差别。todo 通常只描述“要做什么”；Pin 更关心“从哪里继续”。

## 工作方式会随之变化

当记录和交接的成本下降之后，人会开始记录更多以前不值得记录的东西。

这不一定意味着要做更多事。更准确地说，是你可以更便宜地试探和保留选择。

看到一个可能的重构方向，可以先 pin 住，让 agent 以后探索。发现一个 UI 问题，可以附截图，不必打断当前任务。review 里冒出一个边缘问题，可以把它从当前 PR 里拆出来，而不是顺手塞进这次改动。

Pin 让这些小线索有了去处。它们不必马上变成正式 issue，也不必污染当前 agent 的任务边界。

这对并行 agent 尤其重要。多个 agent 同时工作时，最危险的不是它们慢，而是它们把上下文混在一起。Pin 提供了一种低成本的分流方式：当前任务继续走，旁支线索被固定下来，等有空时再交给合适的 agent 或人。

## 代价和边界

Pin 不是越多越好。

如果什么都 pin，它很快会变成另一个垃圾箱。一个健康的 Pin 列表应该是流动的：open 表示还需要关注，done 表示闭环，dropped 表示你有意识地决定不做。

这三个状态都重要。尤其是 dropped。很多工作不值得继续，但值得留下“为什么不继续”的痕迹。否则几周后你会重新发现同一个想法，再浪费一次判断成本。

Pin 也不是正式项目管理系统的替代品。大功能、跨团队计划、必须排期的工作，仍然应该进入 issue 或路线图。Pin 更适合那些介于“现在不做”和“不能丢掉”之间的东西：截图、线索、后续动作、半成熟想法、agent 可以接手的小任务。

## Pin 的本质

AI 编码让开发者的角色发生了变化。你不再只是写代码的人，也是在管理多个 agent、多个 worktree、多个半成品上下文的人。

在这种工作方式下，最稀缺的不是命令行窗口，而是可恢复的上下文。

Pin 的价值就在这里：它把容易散掉的上下文固定到项目旁边，让未来的人或 agent 能够从一个清晰的交接点继续，而不是从聊天历史和终端滚动里重新考古。

它不是收藏夹。

它是项目的工作记忆。
